일별 가입자 통계가 대시보드와 DB 조회 결과가 달랐습니다. 항상 조금씩, 자정 근처에서요.

## 처음 의심한 것들

집계 쿼리를 먼저 봤습니다. 문제없었습니다. 그다음 캐시를 지워 봤습니다. 그대로였습니다.

차이가 나는 데이터를 하나씩 꺼내 보니 규칙이 보였습니다. 어긋나는 건 전부 한국 시간으로 오전 9시 이전에 가입한 사람들이었습니다. 9시간. 익숙한 숫자죠.

## 원인

애플리케이션은 `datetime.now()` 로 시각을 저장하고 있었습니다. 이 함수는 서버의 로컬 시간대를 따르는, 타임존 정보가 없는 값을 돌려줍니다.

```python
from datetime import datetime
datetime.now()
# datetime.datetime(2025, 5, 30, 14, 3, 21, 118273)  ← 어느 지역인지 알 수 없음
```

로컬 개발 머신은 KST, 운영 서버는 UTC였습니다. 같은 코드가 다른 시각을 저장하고 있었던 겁니다.

## 고친 방식

저장은 UTC로 통일하고, 보여줄 때만 KST로 바꾸기로 했습니다.

```python
from datetime import datetime, timezone
from zoneinfo import ZoneInfo

created_at = datetime.now(timezone.utc)          # 저장
created_at.astimezone(ZoneInfo("Asia/Seoul"))    # 표시
```

집계 쿼리에서도 날짜 경계를 KST 기준으로 잘라야 합니다. UTC 기준으로 자르면 한국의 하루와 9시간 어긋납니다.

기존 데이터는 마이그레이션으로 9시간을 더해 UTC로 맞췄습니다. 다행히 서버 타임존이 바뀐 시점이 명확해서 구간을 나눌 수 있었습니다.

## 배운 것

타임존 없는 `datetime` 은 코드에 남겨 두지 않는 게 낫습니다. 지금은 린트 규칙으로 `datetime.now()` 를 아예 막아 뒀습니다.

시간을 다루는 코드는 로컬에서 잘 돌아가는 게 아무 증거도 되지 못합니다.
