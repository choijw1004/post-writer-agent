주문 목록 API가 느리다는 제보를 받았습니다. 처음엔 데이터가 많아져서 그런 줄 알았는데, 아니었죠.

## 증상부터

로컬에서 100건을 불러오는 데 1.8초가 걸렸습니다. 데이터가 100건인데 1.8초면 뭔가 이상합니다. 쿼리 로그를 켜 봤습니다.

```python
import logging
logging.getLogger("django.db.backends").setLevel(logging.DEBUG)
```

로그가 화면을 가득 채웠습니다. 쿼리가 101개 나가고 있었습니다.

## 원인은 N+1

목록을 한 번 가져온 뒤, 각 주문의 사용자 정보를 반복문 안에서 하나씩 조회하고 있었습니다. 주문 100건이면 사용자 조회 쿼리도 100번입니다. 흔히 N+1 문제라고 부르는 그것이죠.

문제가 되는 코드는 이렇게 생겼습니다.

```python
orders = Order.objects.all()[:100]
for order in orders:
    print(order.user.name)  # 여기서 매번 쿼리
```

`order.user` 에 처음 접근하는 순간 ORM이 DB를 다시 다녀옵니다. 코드만 보면 아무 일도 안 하는 것처럼 보인다는 게 이 문제의 고약한 점입니다.

## 해결

`select_related` 를 붙였습니다.

```python
orders = Order.objects.select_related("user")[:100]
```

쿼리가 101개에서 1개로 줄었고, 응답 시간은 1.8초에서 0.12초가 됐습니다.

외래 키가 아니라 역참조나 M2M이라면 `select_related` 대신 `prefetch_related` 를 씁니다. 전자는 JOIN, 후자는 쿼리 두 번으로 나눠서 파이썬에서 합칩니다.

## 남은 이야기

배포 후에 `django-silk` 를 붙여서 쿼리 수를 대시보드에 띄워 두기로 했습니다. 사람이 눈치채기 전에 숫자가 먼저 알려주는 편이 낫습니다.

느린 API를 만나면 일단 쿼리 수부터 세 보세요. 대개는 거기 있습니다.
